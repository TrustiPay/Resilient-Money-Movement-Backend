Prompt to Generate TrustiPay Central Ledger Backend

You are implementing the TrustiPay Central Ledger Backend.

TrustiPay is an offline-first peer-to-peer digital payment prototype where mobile devices exchange transactions offline and later synchronize them with a central ledger settlement server.

The backend must be implemented using:

Python

FastAPI

SQLAlchemy

SQLite (for prototype)

Pydantic

Swagger/OpenAPI documentation automatically generated by FastAPI

The backend must expose clear API contracts and validation logic.

1. System Role

The central ledger acts as the single source of truth for:

transaction settlement

transaction status

balances

fraud detection features

tamper-evident ledger chain

The system must:

Accept transactions from mobile devices

Validate integrity and authenticity

Build fraud detection features

Run fraud detection component (stub/mock allowed)

Approve or reject transaction

Update balances and ledger

Allow mobile devices to poll transaction status

Allow ledger reconciliation

2. Mobile Transaction Payload

Mobile devices send the following JSON object:

{
"tx_id": "string",
"sender_id": "string",
"receiver_id": "string",
"timestamp": "string",
"amount": 0,
"transaction_type": "string",
"device_type": "string",
"network_type": "string",
"phone_number": "string",
"location": "string",
"prev_hash": "string",
"tx_hash": "string",
"signature": "string"
}

Explanation:

Field Description
tx_id unique transaction id generated on mobile
sender_id sender device id
receiver_id receiver device id
timestamp transaction creation timestamp
amount payment amount
transaction_type QR / Bluetooth / WiFi
device_type mobile device model
network_type wifi / cellular / offline
phone_number phone number used
location province or region
prev_hash previous local ledger hash
tx_hash transaction hash
signature sender cryptographic signature 3. Server Generated Fields

The backend must compute these values during validation.

These must not be trusted from the client.

{
"oldbal_sender": 0,
"newbal_sender": 0,
"oldbal_receiver": 0,
"newbal_receiver": 0
}

Computation:

oldbal_sender = balance(sender_id)
newbal_sender = oldbal_sender - amount

oldbal_receiver = balance(receiver_id)
newbal_receiver = oldbal_receiver + amount 4. Fraud Detection Feature Object

5. Central Ledger Table (Single Ledger Model)

There must be one central ledger table containing all transactions including rejected ones.

Table name:

central_ledger

Fields:

Field Description
ledger_index autoincrement primary key
tx_id unique transaction id
sender_id sender device
receiver_id receiver device
amount payment amount
currency default "LKR"
timestamp transaction timestamp
transaction_type QR / Bluetooth
device_type device model
network_type wifi / cellular
phone_number phone number
location sender location
oldbal_sender balance before transaction
newbal_sender balance after transaction
oldbal_receiver receiver balance before
newbal_receiver receiver balance after
prev_hash previous ledger hash
tx_hash computed transaction hash
signature sender signature
status approved / rejected / duplicate / fraud_review
reason_code rejection reason
trace_id processing trace id

Important rule:

Hash chain must only include approved transactions.

Rejected transactions remain in the same table but are not used to extend the chain.

For approved transactions:

prev_hash = last_approved_tx_hash
tx_hash = SHA256(tx_payload + prev_hash)

Rejected transactions:

prev_hash = NULL
tx_hash = SHA256(tx_payload) 8. Validation Pipeline

When a transaction arrives:

Validate schema

Check duplicate tx_id

Recompute tx_hash

Verify signature

Compute balances

Validate sender balance

Build fraud feature object

Call fraud detection component

Approve or reject

Update ledger and balances atomically
